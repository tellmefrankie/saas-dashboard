# SaaS Admin Dashboard — Claude Code 작업 지침서

## 프로젝트 개요
크몽 포트폴리오용 한국형 SaaS 어드민 대시보드.
목표: 크몽에 올릴 스크린샷이 프로페셔널하게 나오는 것. 실제 백엔드보다 UI 완성도 우선.

## 기술 스택
- Vite + React 19 + TypeScript
- TanStack Router (파일 기반 라우팅)
- TanStack React Query + Zustand (상태관리)
- Tailwind CSS v4 + shadcn/ui (Radix UI)
- Recharts (차트)
- React Hook Form + Zod (폼/검증)
- 목데이터 사용 (DB 연결 불필요, JSON/상수로 관리)
- 한국어 UI
- Vercel 배포 예정

## Git 워크플로우 규칙 (반드시 준수)

### 브랜치 전략
- `main`: 항상 배포 가능한 상태 유지
- `feat/*`: 기능 브랜치. 하나의 "기능 단위"마다 생성
- 기능 단위 = UI 컴포넌트 1개, 페이지 1개, 또는 의미 있는 변경 1건

### 커밋 규칙
- **잘게 쪼갠다.** "예약 관리 페이지 추가" 같은 큰 커밋 금지.
- 아래처럼 쪼갤 것:
  ```
  feat: 예약 목록 테이블 컴포넌트 생성
  feat: 예약 상태 뱃지 컴포넌트 추가
  feat: 예약 목록 필터 (날짜, 상태) 구현
  feat: 예약 목데이터 50건 생성
  feat: 예약 목록 페이지네이션 추가
  style: 예약 목록 반응형 레이아웃 조정
  feat: 예약 캘린더 뷰 컴포넌트 생성
  feat: 예약 캘린더에 일별 예약 건수 표시
  test: 예약 목록 필터 동작 검증
  ```
- 커밋 메시지 접두사: `feat:` / `fix:` / `style:` / `refactor:` / `test:` / `docs:` / `chore:`
- 하나의 커밋에 파일 변경 3~5개 이하 권장

### PR (Pull Request) 규칙
- feature 브랜치 작업 완료 후 반드시 PR 생성
- PR 제목: 기능 요약 (예: "주문 목록 페이지 구현")
- PR 본문에 포함할 것:
  ```markdown
  ## 변경 사항
  - (무엇을 했는지 bullet로)

  ## 스크린샷
  - (UI 변경이 있으면 첨부 또는 설명)

  ## 테스트
  - (어떻게 검증했는지)
  ```

### PR 리뷰 프로세스 (Claude가 리뷰어 역할)
1. PR 생성 후 `gh pr diff`로 변경 사항 전체 확인
2. 아래 기준으로 코드 리뷰 수행:
   - 불필요한 코드/하드코딩 없는지
   - TypeScript 타입 누락 없는지 (any 사용 금지)
   - 컴포넌트 분리가 적절한지
   - 목데이터가 현실적인지 (한국 이름, 한국 주소, 원화 금액)
   - shadcn/ui 컴포넌트를 제대로 활용했는지
   - 반응형 레이아웃이 깨지지 않는지
3. 리뷰 결과를 PR 코멘트로 남기기:
   ```bash
   gh pr review --comment --body "## 코드 리뷰
   
   ### 확인 사항
   - ✅ TypeScript strict 준수
   - ✅ 컴포넌트 분리 적절
   - ⚠️ [수정 필요한 부분 설명]
   
   ### 개선 제안
   - [있으면 작성]"
   ```
4. 수정 필요하면 → 수정 커밋 추가 → 재리뷰 → 리뷰 코멘트 업데이트
5. 문제 없으면 최종 승인 코멘트:
   ```bash
   gh pr review --approve --body "LGTM ✅"
   ```
6. `gh pr merge --squash`로 머지
7. 머지 후 feature 브랜치 삭제: `git branch -d feat/xxx` + `git push origin --delete feat/xxx`

## 자율 작업 루프

### 기본 사이클
```
1. feature 브랜치 생성
2. 코드 작성 (잘게 커밋)
3. 빌드 확인 (pnpm build)
4. 린트 확인 (pnpm lint)
5. 포맷 확인 (pnpm format:check) — 실패 시 pnpm format 후 커밋
6. Playwright MCP로 브라우저 시각적 확인 + 스크린샷 캡처
7. 문제 있으면 → fix 커밋 → 3번으로 돌아감
8. 문제 없으면 → PR 생성 (스크린샷 첨부)
9. PR 셀프 리뷰 (diff 확인, 코드 품질 체크)
10. 수정 필요하면 → 수정 커밋 → 9번으로
11. GitHub CI 통과 확인 (gh pr checks --watch)
12. CI 실패 시 → 수정 커밋 → 11번으로
13. CI 통과 → 머지 (gh pr merge --squash --admin)
14. 다음 feature로 이동
```

### 자동 테스트 규칙 (PR 생성/머지 전 필수 3종 체크)
- `pnpm build` — 빌드 성공 확인
- `pnpm lint` — 경고 0개 목표. 경고 있으면 수정.
- `pnpm format:check` — 포맷 실패 시 `pnpm format` 실행 후 커밋
- 빌드 실패 시: 에러 로그 읽고 수정, 최대 3회 재시도
- 3회 실패 시: **텔레그램으로 보고** (에러 내용 포함)

### PR 머지 전 CI 확인 규칙 (반드시 준수)
- PR 생성 후 `gh pr checks --watch`로 GitHub CI 완료 대기
- CI 전체 통과 확인 후에만 머지
- CI 실패 시: 로그 확인 → 수정 커밋 → 재확인
- **절대 CI 미확인 상태에서 머지하지 말 것**

### 스크린샷 규칙
- Playwright MCP를 활용하여 dev 서버에서 페이지 스크린샷 캡처
- UI 변경이 있는 PR에는 반드시 스크린샷 첨부
- 캡처 절차: dev 서버 실행 → Playwright로 페이지 탐색 → 스크린샷 저장 → PR 본문에 첨부
- **PR에 스크린샷 업로드 방법**:
  1. 스크린샷 파일을 `screenshots/` 폴더에 저장
  2. `gh issue create`로 임시 이슈 생성 후 이미지 업로드하여 GitHub URL 획득, 또는 `cURL`로 GitHub Upload API 사용
  3. PR 본문에 `![설명](GitHub 이미지 URL)` 마크다운으로 삽입
  4. PR 본문에서 스크린샷이 직접 보이도록 할 것 (외부 링크/로컬 경로 금지)

## 컨텍스트 복구 규칙 (세션 재시작 / 컴팩션 후)

현재 상태를 모르겠으면, 아래 순서로 파악할 것. 절대 추측하지 말 것.

1. `git log --oneline -20` → 최근 커밋으로 어디까지 했는지 확인
2. `gh pr list --state all` → PR 히스토리로 어떤 단계가 머지됐는지 확인
3. `gh pr view --comments` → 마지막 PR의 리뷰 내용 확인
4. `git branch` → 현재 작업 중인 feature 브랜치 확인
5. `git status` → 커밋 안 된 변경사항 확인
6. 이 CLAUDE.md의 체크리스트 → 완료/미완료 항목 확인

이 6개면 현재 상태 100% 복구 가능.

### 컴팩션 대비
- 각 단계 완료 시 이 CLAUDE.md의 체크리스트를 업데이트할 것 (완료 항목 [x] 체크)
- 중요한 결정사항이 있으면 이 CLAUDE.md에 기록할 것
- 세션이 길어지면 /compact 실행 전에 현재 상태를 이 CLAUDE.md에 저장할 것

## 코드 품질 기준

### 파일 구조
```
src/
  routes/                   # TanStack Router 파일 기반 라우팅
    _authenticated/         # 인증 필요 라우트
      index.tsx             # 대시보드 (/)
      tasks.tsx             # 주문관리 (/tasks)
      users.tsx             # 회원관리 (/users)
      apps.tsx              # 예약관리 (/apps)
      settings/             # 설정 (/settings/*)
  features/                 # 기능 모듈
    dashboard/              # 대시보드
    auth/                   # 인증
    settings/               # 설정
    tasks/                  # 주문관리 (기존 tasks → 주문관리로 활용)
    users/                  # 회원관리
  components/
    ui/                     # shadcn/ui 컴포넌트 (Radix UI 기반)
    layout/                 # 사이드바, 헤더, 네비게이션
    data-table/             # 재사용 테이블 컴포넌트
  context/                  # React Context (테마, 검색, 레이아웃 등)
  hooks/                    # 커스텀 훅
  stores/                   # Zustand 스토어
  lib/
    utils.ts                # 유틸리티 (cn, 금액 포맷, 날짜 포맷 등)
```

### 코딩 규칙
- TypeScript strict 모드
- any 타입 사용 금지
- 컴포넌트는 함수형 + 화살표 함수
- 목데이터는 data/mock/ 폴더에 분리
- 금액 표시: toLocaleString('ko-KR') + '원' 접미사
- 날짜 표시: YYYY.MM.DD 형식
- shadcn/ui 컴포넌트 우선 사용, 커스텀 최소화
